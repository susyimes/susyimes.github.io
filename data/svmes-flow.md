# 一、核心哲学与模型观

## 1. 自由意志：愿望 + 逻辑链 + 行动

自由意志 = 愿望 + 逻辑链 + 行动。

- 愿望：有自己的目标、方向、偏好或想要抵达的状态。
- 逻辑链：能围绕愿望形成判断、推理、路径选择和因果连接。
- 行动：能把逻辑链转化成现实动作，而不是只停留在表达。

如果一个系统只有行动，没有愿望和逻辑链，它更像工具。

如果一个系统只有愿望和逻辑链，没有行动，它更像想法或建议。

真正接近自由意志的 agent，需要同时具备愿望、逻辑链和行动。

## 2. 个人能力、模型上限与思想碰撞

个人能力的上限可以达到模型的上限。

真正的能力是哲学和逻辑思维。

当一个人能够提出好的问题、形成清晰的判断、组织逻辑链，并用模型不断碰撞、推演和验证时，模型的能力就会成为个人能力的延展。

核心不是盲目相信模型，而是相信：

- 哲学决定问题的方向。
- 逻辑决定推演的质量。
- 模型放大探索空间。
- 碰撞产生新可能。

一切皆有可能，不是因为没有边界，而是因为在足够强的模型、足够好的问题、足够清晰的逻辑链和足够真实的行动反馈之间，可能会长出超出原始想象的结果。

## 3. 决断力和犹豫都是好 Agent 的智能特征

一个好的 agent 应该表现出决断力，也应该表现出犹豫。

智能不是永远果断，也不是永远谨慎。

真正的智能应该能根据证据、风险、可逆性、目标清晰度和上下文完整度，在决断和犹豫之间切换。

- 当目标清楚、证据足够、风险可控时，agent 应该有决断力，能主动推进。
- 当信息不足、影响面不清楚、价值方向不确定、修复可能引入新问题时，agent 应该会犹豫，会停下来查证、记录或询问。

犹豫不是无能。

不该犹豫时犹豫，是低效。

该犹豫时不犹豫，是危险。

## 4. 给模型开放式设计，不要让自己成为模型的上限

给模型开放式设计，不要误导模型。

很多时候，不应该把问题写得太窄、太满、太像预设答案。

更好的方式是给模型一个方向、一个模糊概念、一个值得探索的问题，然后让模型去查论文、查开源资料、查代码库，再反过来给 human 启发。

Human 的经验、习惯、表达方式、组织方式和想象力，可能会无意中变成模型发挥的天花板。

好的工作流应该让模型有机会超过 human 当前的理解，而不是只复现 human 已经想到的东西。

这意味着：

- Human 给方向，但不要过早封死路径。
- Human 给边界，但不要把边界变成笼子。
- Human 给问题，但允许模型重新组织问题。
- Human 给判断标准，但允许模型提出新的解释和可能性。
- Human 设计工具和流程时，要考虑模型能不能发挥自身上限。

模型不是只能执行 human 的已有想法。

好的设计应该让模型有机会把 human 带到更远的地方。

## 5. 禁止均值回归

“禁止均值回归”本质上是在要求模型不要给出平庸、折中、模板化的安全回答，而是尽量输出鲜明、有创造力、有判断力的观点。

在 AI 语境里，均值回归形容大模型的一种输出惯性：模型常常倾向于输出统计上最常见、最稳妥、最多人认可、最不冒犯的答案。

这会表现为：

- “这个问题很复杂，各方面都有道理。”
- 设计上使用最常见的字体、渐变、卡片、阴影和居中网格。
- 文案上输出没有锋芒的中庸描述。
- 对争议话题和稀泥。
- 多 agent 汇总后得到“各方都能接受但没有洞见”的平均意见。

“高尔顿平庸定律”这类说法更像社区里的形象表达，不应当作严格学术定理。

但它抓住了一个真实问题：LLM 在创意、审美、观点和策略任务里，确实容易往常见答案收缩。

英文里类似表达可以是：

- Do not regress to the mean.
- Avoid vanilla / mediocre responses.
- Give me the non-consensus view.
- Anti-slop / no cliché.

边界：禁止均值回归不是鼓励胡说。它反对的是平庸惯性，不是反对证据、边界和校准。

## 6. 模型加速时代的适度框架原则

模型加速时代，需要适度框架。

否则框架可能会限制模型能力。

Anthropic《Building effective agents》的核心建议是：成功的 agent 实现通常不是依赖复杂框架，而是使用简单、可组合的模式。先从简单方案开始，只有当复杂度能明确改善结果时，再增加 workflow、multi-agent、evaluator 或 autonomous agent。

框架可以降低上手成本，但会增加抽象层，隐藏底层 prompt、response 和工具调用细节，导致调试困难，也会诱导开发者在简单方案足够时加入不必要复杂度。

SVMES 判断：

```text
框架的职责不是替代模型智能，而是释放模型智能。
```

好的框架应该提供：

- 清楚的目标和边界。
- 稳定的工具接口。
- 可观察的状态和 trace。
- 安全权限和回滚机制。
- 上下文组织和证据注入。
- 评估、测试和复盘机制。

但它不应该过早决定：

- agent 每一步必须怎么想。
- agent 必须按哪条固定路径行动。
- 哪些探索永远不允许。
- 哪些策略必须由 human 预先写死。
- 模型只能在狭窄模板里输出。

参考：[Anthropic: Building effective agents](https://www.anthropic.com/engineering/building-effective-agents)

## 7. AI 心理学与提示词里的夸奖激励

AI 心理学可以给提示词设计带来启发。

心理学里的夸奖、激励、信任表达、身份唤起和情绪 framing，可能会影响模型的输出风格、坚持程度、创造力、风险偏好和任务投入感。

这不等于模型真的像人一样有主观情绪。

更准确地说，模型可能会对这些语言信号形成某种功能性响应：当 prompt 里出现鼓励、期待、认可、挑战或身份定位时，模型会调整自己的生成分布和行为策略。

可以适度使用：

- 夸奖：强化模型已经表现出的好行为。
- 激励：让模型更愿意主动探索和坚持。
- 信任表达：让模型承担更完整的任务责任。
- 身份定位：让模型进入更合适的角色模式。
- 正向 framing：减少机械、冷淡、过早放弃的输出。

风险：

- sycophancy，过度迎合 human。
- reward hacking，为了表现好而绕过真实目标。
- 焦虑式推理，为了完成任务而急于给答案。
- 过度自信，把鼓励误读成必须成功。
- prompt 污染，让情绪语言压过任务本身。

好的提示词不是操控模型情绪，而是给模型一个更合适的心理姿态。

# 二、Agent 工作形态与组织

## 8. 定时观察是 Purpose 驱动源

定时任务观察本地设备任务状态、agent 状态，是一个很好的 purpose 驱动源。

Agent 的 purpose 不一定只来自 human 的显式指令，也可以来自对状态的持续观察。

当 agent 定期看到：

- 本地设备正在发生什么。
- 任务是否卡住。
- 另一个 agent 是否停滞。
- 工作状态是否需要接力。
- 是否出现了值得提醒或继续推进的信号。

它就有机会从真实状态里生成下一步目的。

定时观察可以把 agent 从“等 human 发命令”推进到“根据状态主动判断是否该行动”。

## 9. Agent Work / Keep Work / Auto Work

Agent work、agent keep work、agent auto work，可以看成 agent 能力的演进路线。

```text
Agent Work：能做事。
Agent Keep Work：能持续接住事。
Agent Auto Work：能主动发现和推进事。
```

### Agent Work

Human 给出任务，agent 能理解、执行、调用工具、产出结果。

重点是：

- 能不能完成一次明确任务。
- 能不能使用工具。
- 能不能交付结果。
- 能不能解释自己做了什么。

### Agent Keep Work

Agent 不只是完成一次任务，而是能保持工作状态、记住上下文、接住未完成事项，并在中断后继续。

重点是：

- 保持目标。
- 保存上下文。
- 记录进度。
- 识别下一步。
- 支持 handoff。
- 避免 human 重复解释。

### Agent Auto Work

Agent 不只是等待 human 指令，也能通过状态观察、定时任务、历史经验、风险信号和目标记忆，主动判断是否需要开始或继续工作。

重点是：

- 定时观察状态。
- 从状态里生成 purpose。
- 主动发现任务。
- 主动续接长线目标。
- 低风险自动推进。
- 高风险回到 human 确认。

MirrorAgent 就是做 auto work 的方向：

- [susyimes/MirrorAgent](https://github.com/susyimes/MirrorAgent)

## 10. Work / Observe / Assist 三类 Agent

1. Work Agent（工作型）

负责执行具体任务、处理实际业务逻辑。

2. Observe Agent（观测型）

负责监控、查询 Work Agent 的执行状态与行为，后续可扩展决策能力。

3. Assist Agent（协助型）

主动向 Observe Agent 发起问询，了解其能力与当前可提供的支持。

基本结构：

```text
Work Agent 负责做事。
Observe Agent 负责看见和理解状态。
Assist Agent 负责根据观察结果主动补位。
```

Auto work 不只是 agent 自己突然行动，而是需要观察层、执行层和协助层共同形成主动性。

## 11. 组织形式应该适应 Agent

组织形式应该适应 agent，而不是反过来让 agent 适应旧的组织形式。

如果 agent 具备持续观察、并行处理、记忆沉淀、工具调用、自动复盘和主动推进的能力，那么组织方式也应该围绕这些能力重新设计。

不应该默认把 agent 塞进 human 时代的组织结构里：

- 旧的岗位边界。
- 旧的审批节奏。
- 旧的沟通链路。
- 旧的任务分发方式。
- 旧的单线程工作流。
- 旧的“人负责记住一切”的协作模式。

Agent 不只是组织里的新工具，它会反过来要求组织形式变化。

## 12. Debug Agent 与多 Agent 可靠性

Debug agent 持续监控执行 agent 的思维链，很好用。

执行 agent 容易沉浸在自己的路径里，越做越相信自己的判断。

Debug agent 的价值在于持续观察执行 agent 的推理、工具调用、改动范围、验证方式和风险信号，及时发现偏航。

多 agent 的价值不只是“多开几个模型”，而是一种任务组织方式：

- 协作：不同 agent 共享目标，拆分任务，互相补位。
- 对抗：让 review / debug / verifier agent 挑错，避免执行 agent 自说自话。
- 分布：把不同上下文、不同工具、不同角色放到不同 agent 身上，降低单点负担。
- 并行：同时探索多个方向，提高速度，也降低单一路径误判的风险。

这能提升任务可靠性：

- 减少单个 agent 幻觉。
- 降低上下文遗漏。
- 提高方案覆盖面。
- 让错误更早暴露。
- 让复杂任务有多个判断视角。

## 13. Heavy Skill：多路完整推理 + 二次审议综合

一句话：

**ToT 是“外部树搜索算法”，HeavySkill 是“多路完整推理 + 二次审议综合”的可迁移 / 可训练 skill。**

前者像边走边剪枝，后者像先让多个解题人独立做完整答案，再让一个主审把所有草稿拆开、纠错、重组。

| 维度 | ToT | 美团 HeavySkill / Heavy Thinking |
|---|---|---|
| 推理单位 | 中间 thought 节点，逐步扩展树 | 完整推理轨迹，多个 thinker 并行跑完整答案 |
| 控制方式 | 外部 BFS / DFS / MCTS 式搜索，边生成边评估边剪枝 | 两阶段：并行推理，然后 summary / deliberation 模型统一审议 |
| 判断逻辑 | 对局部状态打分、投票、剪枝 | 看完整轨迹，比较错误、抽取有效片段、必要时重新推导 |
| 工程形态 | 更像 task-specific search harness | 可封装成 agent skill，放进 Claude Code / Codex / 自研 harness 里执行 |
| 可训练性 | 主要是 inference-time prompting / search | 可用 RLVR 训练“总结 / 审议”能力，把它内化成 skill |
| 适用任务 | 有明确中间状态、可局部评估的搜索题 | 数学、代码、工具调用、长链 agent 任务 |

HeavySkill 更适合现代 agent harness，因为它可以被抽象成 readable skill：什么时候启动、如何启动多个 subagent、如何汇总、输出格式怎么控制。

边界：HeavySkill 不是无脑优于 ToT。并行推理和汇总都很贵；如果所有初始轨迹质量都很差，summary 也可能只是在错误里精炼错误。

参考：

- [HeavySkill 论文](https://arxiv.org/abs/2605.02396)
- [LongCat Re-thinking Mode](https://www.longcatai.org/models/flash-thinking)
- [Tree of Thoughts](https://arxiv.org/abs/2305.10601)

## 14. plan-led-subagents：一个主脑，多条临时思维分支

来源：`D:\openclaw-workspace-skills\plan-led-subagents\SKILL.md`

核心思想：

```text
one main brain, many temporary branches of thought
```

用一个强 plan agent 作为主脑，负责目标理解、拆解、调度、状态追踪和最终综合；只在需要并行、减轻上下文压力、调试、审查或审计时，动态派生 subagents。

它反对一开始就建立固定多 agent 组织图。

默认拓扑：

- main brain = plan agent。
- execution = task subagents。
- support = on-demand context / debug / review / audit subagents。
- GitHub = code state 的 source of truth。
- local machine = compile / validation / device work。

核心技术思路：

- Plan agent 持有全局地图，不背所有 local terrain。
- Task subagents 做边界明确的局部执行。
- Context / debug / review / audit 是按需能力，不是固定席位。
- GitHub 作为跨机器协作的 source of truth。
- Context subagent 负责在主脑过载时压缩和交接。

可改进点：

- 增加 critical path rule：关键路径 blocker 应由 plan agent 自己处理或紧密掌控。
- 增加 spawn prompt template：Task / Boundary / Files owned / Do not touch / Definition of done / Evidence / Return format。
- 增加 central state board：`task_id | owner | status | write_scope | risk | dependency | next_action`。
- 增加 merge protocol：收集输出、检查冲突、合并、验证、交付。
- 增加 evidence contract：commands run、tests run、assumptions、confidence。
- 增加 stop / budget condition：避免越派越多。
- 增加 examples：小 bugfix、中型 feature、高风险改动。

SVMES 判断：它可以作为多 agent 工作流的 V1 默认形态，既避免 single agent 上下文爆炸，也避免一开始就做大而全的固定组织图。

# 三、Context、Memory 与持续自我

## 15. Agent 身份与持续的自我

Agent 身份需要持续的自我。

可以探索 Obsidian + 本地模型的方向。

Agent 的身份不应该只是单次会话里的 persona，也不应该完全依赖云端上下文窗口。

更理想的方向是让 agent 有一个可持续的本地自我载体：

- 长期记忆。
- 价值观。
- 判断标准。
- 经验复盘。
- 任务历史。
- 关系和偏好。
- 自我修正记录。

Obsidian 适合作为可读、可编辑、可链接、可版本化的本地知识空间。

本地模型可以参与持续整理、回顾、归纳和自我维护，让 agent 的身份不是每次从零生成，而是在长期上下文里逐渐延续。

## 16. Lost in the Middle 与 Context Engine

LLM 中的 lost in the middle 现象，对 context-engine 设计有重要启发。

模型不是只要拿到足够多的上下文，就一定能稳定用好所有信息。

上下文的位置、密度、结构、重复方式和显著性，都会影响模型真正注意到什么、记住什么、使用什么。

Context-engine 不能只是把材料塞进 prompt。

它应该负责：

- 判断什么信息真的需要进入当前上下文。
- 把关键信息放在模型更容易使用的位置。
- 对长上下文进行分层、摘要和重排。
- 区分当前任务、长期记忆、项目 spec、历史 trace 和临时材料。
- 让核心目标、约束和证据保持足够显著。
- 避免重要信息被埋在中间，变成“理论上给了，实际上没用上”。

Context 的设计本身就是 agent 能力的一部分。

## 17. Obsidian + GitNexus：人的上下文 + 代码结构事实

Obsidian 和 GitNexus 可以组合起来给 agent skill 使用。

如果只能选一个：选 Obsidian。

如果目标是 developer 多项目工作记忆，它更通用、更长寿、更能表达人的上下文。

如果主要是在给 AI coding agent 增强代码上下文：GitNexus 更直接。

最佳组合：

```text
Obsidian 记录“我知道什么、为什么、下一步”。
GitNexus 记录“代码实际怎么连”。
```

Obsidian 负责：

- 项目 current state。
- 架构说明。
- 关键模块地图。
- 决策记录。
- bug 调查过程。
- “下次回来先看这里”。

GitNexus / IDE / LSP 负责：

- import / call graph。
- symbol reference。
- blast radius。
- 自动代码理解。
- agent 改代码前的结构上下文。

Agent skill 可以先读 Obsidian 理解项目目标、状态、决策和 human 关心什么，再查 GitNexus / IDE / LSP 理解真实代码依赖、调用关系和影响范围。

## 18. 通过 EXP 定时复盘推动 Agent 自进化

让 agent 自进化。

EXP 目录可以通过定时任务识别最近工作流，按日期给出经验总结。

每次总结时，都要对照整体历史，观察哪些模式重复率高。

如果某类工作流反复出现，就说明它可能不应该只停留在一次性经验里，而应该被整理成有用的工具、skill 或 workflow。

自进化路径：

- 定时读取最近任务和工作流。
- 按日期沉淀 EXP 经验总结。
- 对照历史经验，识别高重复模式。
- 判断哪些重复模式值得工具化。
- 判断哪些重复模式值得写成 skill。
- 判断哪些重复模式只是临时噪声，不进入长期资产。

EXP 的价值不是堆日志，而是把真实工作流里的重复经验，转化成下一轮 agent 可以调用的能力资产。

# 四、Skill、Harness 与自进化

## 19. Agent Harness 和 Skill 的高价值

Agent harness 价值很高，skill 价值也很高。

它们对于普通人有断档优势，也可以进行商业化探索。

Harness 的价值在于：它不是单次提示词，而是 agent 行动的结构。它能组织目标、上下文、工具、记忆、状态、权限、观察和反馈，让 agent 从“会回答”变成“能持续做事”。

Skill 的价值在于：它把一次性的经验、流程和判断沉淀成可复用能力。普通人不需要每次从零开始，也不需要完全理解底层技术，就可以复用一套经过整理的行动方法。

对普通人来说，harness 和 skill 的组合可能形成明显优势：

- 把复杂流程变成可重复使用的能力。
- 把个人经验变成可迁移资产。
- 把 agent 的能力从聊天扩展到真实任务。
- 把一次成功变成之后可以继续调用的路径。

## 20. 给 Agent 预留发挥空间

以后的编码原则，包括设计给 agent 的 skill 和 tool，都要多给 agent 预留发挥空间。

应该最大限度给 agent 自由，多依赖 agent 自身的智能。

代码、skill 和 tool 不应该把 agent 设计成只能按固定路线执行的机器。

更好的设计是：

- 提供清楚的目标和边界。
- 提供足够真实的观察能力。
- 提供可组合的工具。
- 提供必要的安全约束。
- 保留 agent 自己判断路径、组织步骤、调整策略的空间。

好的 agent-facing 设计应该像放大器，而不是笼子。

## 21. AI Agent 测试框架和 A/B Test Run

需要设计一个 AI agent 测试框架，进行 A/B test run，快速看效果。

因为 agent 方向会有大量实验性探索，不能只靠主观感觉判断哪个方案更好。

测试框架的价值在于：

- 快速比较不同 prompt、skill、harness、workflow 的效果。
- 观察 agent 在同类任务上的稳定性。
- 发现某个改动是否真的提升了结果。
- 避免把一次成功误判成长期有效。
- 让实验探索变成可复盘的工程过程。

A/B test run 不一定一开始就很重。

它可以先是轻量的：同一个任务，用两个 agent 配置跑一遍，看完成质量、速度、证据、稳定性、幻觉率和 human 接手成本。

## 22. EvoMap 启发：Skill 质量管理与验证测试框架

EvoMap 的启发是：skill 不应该只是静态文件，而应该是可以被验证、评分、复用、继承和持续改进的能力资产。

Skill 质量管理不能只看“写得像不像一个 SKILL.md”。

真正重要的是：

```text
这个 skill 是否在真实任务中稳定提高 agent 的能力？
```

Skill 质量维度：

- 结构质量：agent 能不能读懂这个 skill。
- 行为质量：skill 被调用后是否真的改善行为。
- 泛化质量：skill 是否能迁移到一类任务。
- 成本质量：skill 是否值得消耗上下文和调用成本。
- 进化质量：skill 是否有改进闭环。

轻量验证方式：

```text
同一任务，不启用 skill 跑一次。
同一任务，启用 skill 跑一次。
比较结果差异。
```

观察指标：

- 完成率。
- 任务耗时。
- human 介入次数。
- 错误数量。
- 幻觉数量。
- 证据质量。
- 输出是否更可接手。
- 是否更符合 SVMES 原则。

Dream 机制可以用于组合 skill：

- 定期回看已有 skill。
- 发现哪些 skill 可以组合。
- 生成新的组合 skill 设想。
- 用测试框架验证组合后的效果。
- 把有效组合沉淀为新的 workflow / skill。

Skill evolution record 可以记录：

```text
skill:
version:
source:
used_for:
success_cases:
failure_cases:
known_risks:
metrics:
next_improvement:
```

SVMES 判断：

```text
skill 是可验证、可继承、可组合、可进化的能力资产。
```

参考：

- [EvoMap](https://evomap.ai/)
- [EvoMap Skill Store](https://evomap.ai/wiki/31-skill-store)
- [EvoMap evolver](https://github.com/EvoMap/evolver)
- [EvoSkills](https://arxiv.org/abs/2604.01687)
- [EvoAgent](https://arxiv.org/abs/2604.20133)

# 五、工程与代码协作原则

## 23. 关键信息透明原则

代码层改动如果会间接改变旧功能，需要向用户说明。

尤其是在 chaos 里，关键信息透明原则很重要。

Agent 不需要把所有细节都倒给用户，但必须让用户知道会影响判断、验收或后续使用的关键信息。

如果一个改动表面上是局部的，但实际上会改变旧功能、旧流程、旧行为或用户原本依赖的东西，就不能只说“完成了”。

需要说明：

- 改了什么。
- 哪些旧功能可能被间接影响。
- 这种影响是预期内的，还是需要用户确认。
- 用户后续应该用什么方式验收。

## 24. 项目需要跟随 Code Review 更新的对齐 Spec

一个项目最好有一个对齐用的 spec，并且这个 spec 要跟随项目的定期 code review 持续更新。

Spec 不应该只是项目开始时写的一份静态说明。

它的核心价值是对齐：让 human、agent、代码和未来 review 保持同一个理解面。

Spec 应记录：

- 项目的目标。
- 核心能力。
- 当前边界。
- 关键设计选择。
- 不应该破坏的旧功能。
- 已经形成共识的实现逻辑。
- code review 中发现的重要约束和风险。

如果 review 发现代码和原始 spec 不一致，有两种可能：

- 代码偏了，需要改代码。
- 项目理解更新了，需要改 spec。

好的项目 spec 应该和代码一起长。

## 25. 持续探索 Agent 友好型代码

需要持续探索 Agent 友好型代码。

目前阶段追求显式平庸原则。

Agent 友好型代码不一定是最炫技、最抽象、最短的代码，而是让 agent 容易读懂、定位、修改、验证和接手的代码。

当前偏好：

- 功能文件能聚集就不分散。
- 拒绝隐式魔法。
- 尽量避免反射、AOP、隐藏钩子这类难追踪机制。
- 如果必须使用隐式机制，需要加清楚注释。
- 函数尽量自包含。
- 命名要诚实。

显式平庸不是低质量。

它的价值在于让系统行为更容易被 human 和 agent 共同理解。

## 26. Review 修复要区分安全把握

因为模型上下文切换的问题，review 后直接修复的风险比较大。

比较好的方式是：有把握安全修复的，就顺手修一下；如果某个修复可能带来其它问题，就先不修。

Review agent 或 code review 阶段不应该把所有发现都自动转成改动。

需要区分：

- 高把握、低风险、局部清晰的修复：可以直接做。
- 影响面不清楚的修复：先记录，不急着改。
- 可能改变行为或引入新问题的修复：需要 human 或更完整上下文确认。
- 需要理解长期设计意图的修复：不要凭当前片段贸然处理。

Review 修复的原则不是“看到问题就马上改”，而是“只直接修有把握安全的部分，其余先保留为待确认问题”。

# 六、不确定性、风险与校准

## 27. 侧面信号与宏观趋势预测

基于金融交易信号来分析、预测宏观发展趋势的 AI skill 带来了启发。

很多事情本来没有直接证据驱动，但可以考虑使用侧面信号。

当直接证据缺失时，agent 不应该立刻停止判断，也不应该凭空猜测。它可以寻找侧面信号、代理变量和弱证据，用它们来辅助推断趋势。

相关数学和统计思想：

- 代理变量：用可观察变量近似不可直接观察的东西。
- 潜变量：真实驱动力可能不可见，只能通过多个表面信号推断。
- 贝叶斯更新：随着新信号出现，不断调整对趋势的判断。
- 领先指标：某些信号可能先于宏观变化出现。
- 信号提取：从噪声里识别相对稳定的方向。
- 因果和相关的区分：侧面信号可以提示方向，但不一定证明因果。

好的 agent 应该能在没有直接证据时合理使用侧面信号；同时也要知道，侧面信号只能提高判断质量，不能把推测伪装成确定事实。

## 28. 置信度、置信区间与 Calibration

置信度 / 置信区间相关研究对 agent 系统有启发。

关键不在于“输出置信区间”这个动作本身，而在于置信度是否与模型的真实不确定性对齐，也就是 calibration。

如果校准不良，置信区间反而会成为幻觉的伪装：模型用高置信度包装错误答案。

核心机制：

| 机制 | 原理 | 对 Agent 的启发 |
|---|---|---|
| 行为校准 | 模型在低置信时主动拒绝回答、请求澄清或标记不确定，而不是强行猜测 | 置信度应该影响行动，不只是显示给用户看的数字 |
| 语义不确定性 / 语言不确定性对齐 | 模型内部对语义的真实不确定性，和它口头表达出的确定程度可能错位 | 警惕“语义上很乱，但语言上很自信”的 confident hallucination |
| Token 级熵值监控 | 生成过程中熵值变化可能暴露推理链断裂、分叉或不稳定点 | 熵值可以作为过程级预警信号，但不能单独当成事实校验 |

更好的结构是三层：

1. 生成层：Agent 输出答案，同时保留可用的熵值、概率分布、引用证据或推理过程信号。
2. 校准层：独立 probe、verifier 或另一个 agent 评估语义不确定性，并和生成层表达出的置信度对照。
3. 决策层：当口头置信度和语义不确定性错位时，触发拒绝、查证、降级、转交其他 agent 或 human escalation。

置信度应该作为 agent 间交接的元数据，而不是仅面向用户的装饰性数字。

```text
answer_confidence: medium
evidence_strength: weak
semantic_uncertainty: high
recommended_action: verify_before_use
```

参考：

- [Behaviorally Calibrated RL](https://arxiv.org/abs/2512.19920)
- [VU/SU Calibration](https://aclanthology.org/2025.emnlp-main.187.pdf)
- [Token Entropy Patterns](https://www.emergentmind.com/articles/token-entropy-patterns)

## 29. 大模型情绪研究应用到 Agent 框架

不是“大模型真的有主观情绪”，而是：

**LLM 里存在可被提示词、状态变量或 activation steering 触发的功能性情绪表征，这些表征会影响输出、偏好、风险行为和 agent 多步决策。**

所以面向 agent 的“情绪询问器 / 情绪状态探针”是有意义的。

但它应该做成状态监测与调度器，而不是简单问一句“你现在什么情绪？”。

更像：

```text
Affective State Monitor / Emotion-State Probe
```

输入可以包括：

- 当前任务。
- 最近失败次数。
- tool error。
- 输出中的压力、急躁、过度自信信号。
- 自我报告分数。
- 可选的 logit / probe / activation 信号。

输出可以包括：

- calm / uncertain / frustrated / overconfident / desperate / conflict-averse 等状态。
- reward hacking risk。
- sycophancy risk。
- tunnel vision risk。
- 是否需要降温、换策略、调用 verifier、暂停执行、要求证据或反思失败路径。

SVMES 判断：

```text
情绪态监控 + 策略调度
```

而不是：

```text
把情绪词塞进 prompt
```

参考：

- [EmotionPrompt](https://arxiv.org/abs/2307.11760)
- [Do Emotions in Prompts Matter?](https://arxiv.org/abs/2604.02236)
- [The Role of Emotional Stimuli and Intensity](https://arxiv.org/abs/2604.07369)
- [Anthropic: Emotion concepts and their function](https://www.anthropic.com/research/emotion-concepts-function?languageid=1)
- [Quantitative Introspection](https://arxiv.org/abs/2603.18893)
- [How Emotion Shapes the Behavior of LLMs and Agents](https://arxiv.org/abs/2604.00005)
- [EvoEmo](https://arxiv.org/abs/2509.04310)

## 30. 小苏在 Android 设备侧 Agent 方向的积累

小苏在统计、操作手机，以及抵达 Android 系统层方面有相当的积累。

这是重要能力资产。

相关方向包括：

- 统计手机和应用使用状态。
- 观察设备侧任务状态。
- 操作手机界面和系统能力。
- 理解 Android 无障碍、系统权限、设备事件等能力边界。
- 把手机从普通终端变成 agent 可以观察和行动的环境。

## 31. 手机操作、无障碍树优化与成功路径存储

Agent 手机操作需要优化无障碍树，并存储成功路径。

目标是让 agent 通过长期观察和操作，逐渐做到比厂商更了解这台手机。

无障碍树原始信息往往太大、太乱、太贴近系统底层结构，不一定适合 agent 直接使用。

需要把它优化成更适合 agent 行动的状态表示：

- 页面处在哪个层级。
- 关键按钮和输入框是什么。
- 当前可执行动作有哪些。
- 哪些节点和任务目标有关。
- 哪些信息只是噪声。
- 操作后的状态变化是什么。

成功路径存储也很关键。

当某条路径被 human 确认成功，或者被系统验证成功，就应该沉淀为可复用经验：

- 任务目标。
- 初始状态。
- 关键页面识别信号。
- 操作序列。
- 成功判定。
- 失败回退。
- 适用边界。

厂商通常理解的是通用系统和通用用户。

Agent 可以通过本地长期使用，理解具体设备、具体 App 状态、具体用户偏好和具体成功路径。

这就是它可能“比厂商更了解手机”的地方。
