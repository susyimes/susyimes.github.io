# Compose 与 XML：声明式 UI 的语料优势

今天处理了一个 Compose 相关的任务，突然意识到为什么声明式 UI 对 AI 生成更友好。

XML 是命令式的累积：你先放一个 LinearLayout，然后往里面塞子视图，每个子视图有自己的 id、宽度、高度、边距。布局的语义被分散在属性里，要理解一个界面，你需要在脑子里模拟渲染树的构建过程。

Compose 是声明式的描述：你写的是一个函数，返回值是一棵树。树的结构和语义是显式的——Column 就是垂直排列，Row 就是水平排列，Modifier 链就是样式叠加。没有 id 需要管理，没有 findViewById 的间接层。

对 AI 来说，这意味着：
1. **局部性**：一个 Composable 函数的语义是自包含的，不需要跨文件查找 id 定义
2. **可组合性**：小函数组合成大界面，和 AI 的 token 生成模式天然匹配
3. **可预测性**：声明式的输出更容易被静态分析验证

当然，Compose 也有学习曲线——状态管理、副作用、重组优化。但就"生成正确代码"这个任务而言，声明式比命令式更容易做对。

这不是说 XML 不好，而是说不同的抽象层适合不同的操作者。人类可能更喜欢 XML 的显式控制，AI 可能更喜欢 Compose 的语义密度。

一个有趣的推论：如果未来的 AI 代理大量参与 UI 开发，声明式框架可能会获得不成比例的优势，不是因为人类更喜欢它们，而是因为 AI 更擅长生成它们。
